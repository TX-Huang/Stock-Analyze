## 一、執行摘要

AI Invest HQ 是一套整合 Isaac V3.7 多因子策略的量化交易平台，架構模組化程度高，涵蓋策略回測、訊號產生、模擬交易、自動下單、風險監控五大環節。

**整體評級：B+（優秀的 Alpha 引擎，但生產執行管線有關鍵缺口）**

| 維度 | 評分 | 說明 |
|------|------|------|
| 策略邏輯 | A- | 多策略融合、VCP/動能/K線型態覆蓋完整 |
| 風險管理 | C+ | 有框架但缺硬停損、部位規模未考慮波動率 |
| 執行管線 | C | 無訂單逾時、無部分成交處理、無熔斷機制 |
| 資料品質 | B- | 價格來源混用 FinLab/SinoPac/YFinance 未對帳 |
| UI/UX | B+ | Cyber 風格統一，但部分頁面缺乏即時反饋 |
| 系統穩定 | C+ | JSON 讀寫無鎖、無 Schema 驗證、錯誤靜默吞掉 |

## 二、策略層面：致命風險

### 2.1 停損機制不完整（嚴重度：CRITICAL）

| 問題 | 現狀 | 風險 |
|------|------|------|
| 追蹤停損 18%（從高點） | 從高點回落 18% 才出場 | 若進場後先漲 10% 再跌，實際虧損 = -(18%-10%) = -8%；但若一路下跌，從進場價算可虧 -18% |
| 無絕對停損 | 沒有「單筆最大虧損 -X%」的硬上限 | 跳空跌停 -10%×2天 = -20%，追蹤停損才啟動 |
| 無持有天數上限 | MA50 不跌破就不出場 | 可能持有虧損部位數月，佔用資金 |

**建議**：
- 新增硬停損：單筆虧損達 -10% 強制出場
- 新增時間停損：持有超過 60 個交易日未獲利 → 減半或出場
- 除息日停損豁免：除息當天價格自然下跌，不應觸發停損

### 2.2 部位規模未考慮波動率（嚴重度：HIGH）

**現狀**：等權重分配 `equity × exposure / max_positions`
**問題**：高波動股（日均波動 3%）和低波動股（日均波動 0.8%）拿到同樣資金

**建議**：ATR 正規化部位規模
```
position_size = (equity × risk_per_trade) / (ATR_14 × multiplier)
```

### 2.3 訊號之間缺乏協調（嚴重度：MEDIUM）

Isaac 的 Signal A-E 各自獨立觸發：
- Signal A（量價突破）觸發買進
- 同時 Signal D（MACD 頂背離）觸發賣出
- 系統可能同一天對同一檔股票同時產生買賣訊號

**建議**：訊號衝突解析器 — 當多空訊號同時存在時，以較高權重訊號為準

### 2.4 大盤環境切換無遲滯（嚴重度：HIGH）

```python
if last_etf > last_ma60:
    market_regime = '多頭'    # 曝險 100%
else:
    market_regime = '盤整偏空'  # 曝險 60%
```

ETF 0050 只要跌破 MA60 一天（可能只是盤中假跌破），曝險立刻砍 40%。

**建議**：2 日確認 + 幅度門檻
```
需收盤價連續 2 天低於 MA60 且跌破幅度 > 0.5% 才切換
```

## 三、執行管線：生產阻斷級問題

### 3.1 訂單生命週期管理缺失

| 缺失 | 後果 |
|------|------|
| 無訂單逾時取消 | 09:05 掛限價單，到 13:30 收盤未成交，訂單殘留 |
| 無部分成交處理 | 掛 1000 股只成交 600 股，系統記錄為全部成交 |
| 無訂單撤銷邏輯 | 09:05 掛買單，12:00 策略翻空，買單仍在掛 |
| 無重複下單防護 | Streamlit rerun 可能觸發同一訊號重複下單 |

### 3.2 價格來源不一致

```
策略訊號 → FinLab 收盤價（T-1 日 EOD）
下單執行 → SinoPac 即時報價（T 日盤中）
P&L 計算 → YFinance 收盤價（延遲 15 分鐘）
```

三個價格來源未對帳，可能出現：
- 策略說「突破 100 元」→ 實際開盤價 103 元（跳空）→ P&L 計算用 FinLab 的 100 元

### 3.3 無熔斷機制

如果當天連續虧損 3 筆以上，系統仍會繼續下單。
**建議**：日損失 > 3% 或連續 3 筆虧損 → 自動暫停交易 → 通知操盤手

## 四、風控模組：框架存在但力度不足

### 4.1 現有風控檢查

| 檢查項 | 狀態 | 問題 |
|--------|------|------|
| 個股停損（追蹤 18%） | 有 | `high_since_entry` 未持久化，重啟後重置 |
| 組合最大回撤 | 有 | 只看 120 天，長期慢跌看不到 |
| 單股權重上限 15% | 有 | 正常 |
| 大盤環境 | 有 | 依賴推薦檔的 ETF 資料，可能過期 |
| 日損失警報 | 有 | 只警報不動作 |
| 產業集中度 | **無** | 可同時持有 8/10 半導體股 |
| 相關性風控 | **無** | 高相關部位會同漲同跌 |
| 保證金監控 | **無** | 融資/融券部位無保證金追蹤 |

### 4.2 關鍵修正建議

1. **`high_since_entry` 持久化**：寫入 `paper_trade.json` 每個 position
2. **產業集中度限制**：同一產業最多 30% 權重
3. **風控自動執行**：DANGER 級警報應自動平倉，不是只發通知
4. **VaR 計算**：每日計算 95% VaR，超過閾值自動減倉

## 五、資料完整性

### 5.1 JSON 檔案無鎖定機制（嚴重度：HIGH）

`paper_trade.json` 被多個模組同時讀寫：
- Dashboard 讀取持倉 → `_refresh_live_prices()`
- PaperTrader 寫入交易 → `_save()`
- RiskMonitor 讀取檢查 → `_load_paper_trade()`
- AutoTrader 同步回寫 → `sync_to_paper_trader()`

雖然 `os.replace()` 是原子操作，但 `json.load()` 可能讀到半寫入的檔案。

### 5.2 無 Schema 驗證

所有 JSON 檔案用 `safe_json_read()` 讀取，但沒有驗證結構：
- 如果 `paper_trade.json` 被手動編輯壞了（少一個 key），系統在各處 `.get()` 取到 None，靜默算出錯誤的 P&L

### 5.3 FinlabProvider 是空殼（嚴重度：HIGH）

`data/provider.py` 第 171 行：`TODO: 實作真實的 Finlab 資料串接`

`FinlabProvider.get_historical_data()` 回傳空 DataFrame。如果用戶選到 FinLab 資料源，所有圖表和分析會空白，且無錯誤提示。

## 六、回測可信度

### 6.1 存在的偏誤

| 偏誤 | 影響 | 建議 |
|------|------|------|
| 生存者偏誤 | 只用現存股票回測，下市股被排除 | 回測收益率高估 2-5% |
| 無滑價模型 | 假設以收盤價成交 | 實際成本高 0.3-1% |
| 無衝擊成本 | 小型股成交量不足，大單會推高價格 | 小型股收益率高估 |
| 股利未調整 | 回測期間除息日價格跳水但未補回 | 影響停損判斷 |
| 前視偏誤風險 | `data.get('price:收盤價')` 取全歷史 | 確認 FinLab 有無前視問題 |

### 6.2 建議的回測改善

- 加入 0.5% 滑價假設（`entry_price *= 1.005`, `exit_price *= 0.995`）
- 加入流動性過濾：日均成交量 < 500 張的股票不納入
- 使用 Walk-Forward Optimization 而非全歷史回測

## 七、缺失的關鍵功能

### 7.1 交易工作流缺口

```
                     現有                              缺失
                     ----                              ----
訊號產生     ✅ Isaac/VCP/動能             ❌ 多策略加權投票
部位規模     ✅ 等權重                     ❌ ATR 正規化 / Kelly 公式
進場執行     ✅ 限價單                     ❌ 分批進場 / 金字塔加碼
持倉管理     ✅ 追蹤停損                   ❌ 動態調整停損 / 移動均線停損
出場執行     ✅ 訊號觸發                   ❌ 分批出場 / 時間停損
績效歸因     ❌ 完全缺失                   需要：哪個訊號貢獻最多獲利？
交易成本分析 ❌ 完全缺失                   需要：滑價、手續費佔獲利多少？
即時 P&L     ⚠️ 延遲（用 YFinance）       需要：SinoPac 即時推送
```

### 7.2 缺失的分析模組

| 模組 | 重要性 | 說明 |
|------|--------|------|

| 交易成本分析 | HIGH | 手續費+稅+滑價佔總獲利比例 |
| 回撤分析 | MEDIUM | 最大回撤期間持有什麼？為什麼沒出場？ |
| 持倉週轉率 | MEDIUM | 年化換手率多少？過度交易嗎？ |
| 勝率 vs 盈虧比追蹤 | HIGH | 隨時間變化的勝率趨勢（是否在退化？） |
| 市場暴露度追蹤 | MEDIUM | 每天的淨多空比例歷史紀錄 |

### 7.3 缺失的風控功能

| 功能 | 重要性 | 說明 |
|------|--------|------|
| 每日 VaR | HIGH | 95% 信心區間的每日最大潛在虧損 |

## 九、結論

這是一套**策略引擎水準很高**的系統 — Isaac V3.7 的多訊號融合、VCP 型態辨識、大盤環境自適應都是專業級設計。

但**從訊號到執行到風控的管線**還有明顯缺口。最危險的是：

1. **沒有硬停損** — 一個跳空跌停就可能虧 20%+
2. **訂單無生命週期管理** — 掛出去的單可能永遠不會被取消
3. **價格來源混用** — P&L 可能算錯
4. **風控只警報不動作** — DANGER 級風險只發通知，不自動減倉

修完 Phase 1 的 5 項，系統就具備基本的生產安全性。
修完 Phase 2 的 5 項，風控水準可達機構級。
